How this roster works

Primary character database on disk: rh-wcl-character-links.json — same data as GET /api/wcl/guild/:guildId/characters. Attendance % uses the latest WCL_ATTENDANCE_RECENT_RAIDS logs (default 6), 25-player raids only (Karazhan and Zul'Aman excluded); Events roster badges read that same leaderboard. Heuristic merge still scans broader logs via RH_WCL_LINK_WCL_REPORT_DETAILS. For Raider.io class/spec on the Events roster, the server tries character names in order: first Warcraft Logs name (mapped main), then Raid Helper’s parsed character field, then the signup display name (realm stripped). The first name that returns a profile wins; rioProfileLookupName on the API reflects that hit. With Battle.net credentials set, it also reads Battle.net character profile for English class (and sometimes active spec) when Raid Helper’s class column is wrong or Raider.io misses. Column 1 is only Raid Helper signup names (not log names). Column 2 = Warcraft Logs names (main + alts), comma-separated, or use Add alt under that row to append one log name and save when the Raid Helper name is filled. If the log name matches the signup, column 2 can stay empty.

Heuristic merge uses only Raid Helper signup names for column 1 (recent posted events; env RH_WCL_LINK_RAID_HELPER_EVENTS) and Warcraft Logs characters from recent reports (RH_WCL_LINK_WCL_REPORT_DETAILS). There is no “log names only” fallback — if Raid Helper returns no signups, fix API keys / events. Loose matches may appear as Best-effort. Review chips, then Save all rows.
